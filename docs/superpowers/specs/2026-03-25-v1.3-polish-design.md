# InkPulse v1.3 — Polish & Unification

> Design spec | 25 Mar 2026 | Target user: AI Automation Specialist Orchestrator
> Reviewed: spec-reviewer pass 1 → 7 issues → 6 fixed (Issue 2 disproven by JSONL evidence)

## Goal

Four surgical interventions to make InkPulse a real operational tool for someone orchestrating 4-6 Claude Code sessions across multiple business pillars simultaneously. Zero new features. Zero scope creep.

---

## Intervention 1: Pillar Identity Mapping

### Problem

`PillarInfo.from(cwd:)` uses the last path component with generic capitalization. An orchestrator running 4 pillars sees "Btc_predictions", "Aurahome", "Claude_voice" — meaningless at a glance.

### Solution

Add a known-pillars lookup table in `PillarInfo` that maps directory paths to named pillars with dedicated colors.

```swift
private static let knownPillars: [(pathContains: String, info: PillarInfo)] = [
    ("btc_predictions",             PillarInfo(name: "BTC Bot",    color: Color(hex: "#00d4aa"), shortName: "BT")),
    ("projects/aurahome",           PillarInfo(name: "AuraHome",   color: Color(hex: "#FFD700"), shortName: "AH")),
    ("Astra Digital",               PillarInfo(name: "Astra",      color: Color(hex: "#4A9EFF"), shortName: "AD")),
    ("claude_voice",                PillarInfo(name: "Astra OS",   color: Color(hex: "#A855F7"), shortName: "OS")),
    ("projects/InkPulse",           PillarInfo(name: "InkPulse",   color: Color(hex: "#00d4aa"), shortName: "IP")),
]
```

`from(cwd:)` checks `knownPillars` first (path contains match). Falls back to current last-component logic for unknown dirs.

### Config override

Add optional `pillar_overrides` to `config.json` so any user can define their own mappings:

```json
{
  "pillar_overrides": {
    "my-project": { "name": "My Project", "color": "#FF6B6B", "short": "MP" }
  }
}
```

The `pathContains` key matches against the full cwd path. Config overrides take precedence over hardcoded pillars.

### Files

| File | Change |
|------|--------|
| `UI/SessionRowView.swift` | Add `knownPillars` table, update `PillarInfo.from(cwd:)` |
| `Config/ConfigLoader.swift` | Add `pillar_overrides` parsing |

### Visual result

Before: all cards same blue, generic names.
After: BTC Bot (teal), AuraHome (gold), Astra (blue), Astra OS (purple) — instant pillar recognition.

## Intervention 2: Fix activeTaskName

### Problem

`activeTaskName` is declared in `SessionMetrics` but never written. The field stays nil. An orchestrator can't see WHAT each agent is working on — only the last tool used.

### Root cause

Task information comes from `assistant` events where content blocks include `TaskCreate` or `TaskUpdate` tool_use calls with a `subject` parameter in `input`.

### JSONL evidence (verified)

```json
{
  "type": "assistant",
  "message": {
    "content": [{
      "type": "tool_use",
      "id": "toolu_014583fPd4Aaze1Cmczgq9uG",
      "name": "TaskCreate",
      "input": {
        "subject": "Fix kasse performance — rimuovi @import bloccante",
        "description": "...",
        "activeForm": "Analizzando performance kasse"
      }
    }]
  }
}
```

This is a real `tool_use` content block — NOT a system event. The existing parser in `JSONLParser.parseAssistant()` already iterates `contentBlocks` and creates `ToolUseInfo` for each `tool_use` block.

### Solution

**Step 1**: Add `subject` field to `ToolUseInfo` in `Parser/ClaudeEvent.swift`:

```swift
struct ToolUseInfo {
    let id: String
    let name: String
    let target: String?
    let subject: String?  // NEW: from TaskCreate/TaskUpdate input.subject
}
```

**Step 2**: Extract `subject` in `Parser/JSONLParser.swift`, inside the `tool_use` block parsing (line 114-119). After `extractToolTarget`, add:

```swift
let subject: String?
if toolName == "TaskCreate" || toolName == "TaskUpdate" {
    subject = (block["input"] as? [String: Any])?["subject"] as? String
} else {
    subject = nil
}
toolUses.append(ToolUseInfo(id: toolId, name: toolName, target: target, subject: subject))
```

**Step 3**: In `SessionMetrics.ingest()`, within the `.assistant` case, after existing tool tracking (line 98-101):

```swift
// Task name tracking
for tool in msg.toolUses {
    if (tool.name == "TaskCreate" || tool.name == "TaskUpdate"),
       let subject = tool.subject {
        activeTaskName = subject
    }
}
```

### Files

| File | Change |
|------|--------|
| `Parser/ClaudeEvent.swift` | Add `subject: String?` to `ToolUseInfo` |
| `Parser/JSONLParser.swift` | Extract `subject` from input when tool is TaskCreate/TaskUpdate (line ~118) |
| `Metrics/SessionMetrics.swift` | Write `activeTaskName` from task tool_use blocks |

### Visual result

Agent card activity section shows: `"Fix kasse performance"` instead of just `"Edit -> config.py"`.

## Intervention 3: Dynamic ScrollView Height

### Problem

Popover ScrollView is hardcoded to `frame(height: 350)`. With 1-2 sessions it wastes space. With 5-6 it's cramped. MenuBarExtra popovers ignore `minHeight` so a fixed frame was used as workaround.

### Solution

Replace fixed height with a computed height based on agent count, clamped between min and max. Account for expanded detail panel.

```swift
private var agentsScrollHeight: CGFloat {
    let cardHeight: CGFloat = 120  // estimated per card row (2 cards per row)
    let rows = ceil(Double(snaps.count) / 2.0)
    let expandOffset: CGFloat = expandedSessionId != nil ? 180 : 0
    let computed = CGFloat(rows) * cardHeight + 20 + expandOffset
    return min(max(computed, 180), 500)  // clamp: 180..500
}
```

Apply as `.frame(height: agentsScrollHeight)` in PopoverView.

LiveTab already uses `.frame(maxHeight: 500)` which is fine — leave it.

### Files

| File | Change |
|------|--------|
| `UI/PopoverView.swift` | Replace `frame(height: 350)` with computed height |

## Intervention 4: Shared DashboardViewModel

### Problem

PopoverView and LiveTab each compute the same derived stats: `snaps`, `health`, `totalCost`, `avgCacheHit`, `avgErrorRate`, `peakTokenMin`, `avgTokenMin`, `uptimeMin`, `throughputPerAgent`, `totalAgents`. Any bug fix or new metric must be applied twice.

Note: `avgContextPercent` and `config` are LiveTab-only. They'll be included in the ViewModel for consistency but are currently only consumed by LiveTab.

### Solution

Extract a plain struct with computed properties that derive from `AppState`. Since all source data comes from `AppState`'s `@Published` properties (`metricsEngine`, `tokenHistory`), SwiftUI will re-evaluate the computed properties whenever AppState publishes changes.

```swift
struct DashboardStats {
    let appState: AppState

    var snaps: [MetricsSnapshot] {
        appState.metricsEngine.sessions.values
            .sorted { $0.lastEventTime > $1.lastEventTime }
    }
    var health: Int { appState.metricsEngine.aggregateHealth }
    var totalCost: Double { snaps.map(\.costEUR).reduce(0, +) }
    var avgCacheHit: Double { ... }
    var avgErrorRate: Double { ... }
    var peakTokenMin: Double { appState.tokenHistory.max() ?? 0 }
    var avgTokenMin: Double { ... }
    var uptimeMin: Double { ... }
    var throughputPerAgent: Double { ... }
    var totalAgents: Int { snaps.map(\.subagentCount).reduce(0, +) }
    var avgContextPercent: Double { ... }  // LiveTab-only consumer
    var config: InkPulseConfig { ConfigLoader.load() }  // LiveTab-only consumer
}
```

PopoverView and LiveTab both create a `DashboardStats(appState: appState)` as a local computed property. Their private computed properties are deleted and replaced with `stats.totalCost`, `stats.health`, etc.

### Scope boundary

- Only extract computed stats. Do NOT merge the views.
- `expandedSessionId` stays as `@State` in each view — it is local view state, not shared.
- PopoverView keeps its config toggle, quit button, actions bar.
- LiveTab keeps its budget bar, footer, trend arrows.

### Files

| File | Change |
|------|--------|
| **NEW** `UI/DashboardStats.swift` | Shared computed stats struct |
| `UI/PopoverView.swift` | Replace private computed properties with DashboardStats |
| `UI/LiveTab.swift` | Replace private computed properties with DashboardStats |
| `UI/TabbedDashboard.swift` | No change needed — appState flows through unchanged |

### Why not ObservableObject/Observable?

`AppState` is already `@ObservableObject` with `@Published` properties. A wrapper class would either:
- Use `@Observable` (Observation framework) — won't propagate `@Published` changes correctly
- Use `ObservableObject` — computed properties don't trigger view updates

A plain struct with computed properties is simpler: SwiftUI re-evaluates them when the view body is recomputed (triggered by AppState's `@Published` changes). No observation gymnastics needed.
